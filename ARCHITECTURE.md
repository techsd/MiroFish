# Архітектура системи (System Architecture)

Система MiroFish побудована за принципами мікросервісної та багатоагентної архітектури. Вона розділена на два основні рівні: Frontend (клієнтська частина) та Backend (серверна частина та логіка ШІ).

## Високорівнева діаграма (C4 Context)

1. **Користувач** -> Взаємодіє з Web UI.
2. **Web UI (Vue 3)** -> Надсилає запити до API та отримує потоки даних (SSE) для відображення симуляції в реальному часі.
3. **Backend API (Flask/Python)** -> Обробляє запити, управляє станом симуляцій та оркеструє агентів.
4. **LLM Provider (OpenAI/Ollama/Qwen)** -> Забезпечує інтелектуальну складову (генерація текстів, логіка агентів).
5. **Memory & Graph Database (Zep)** -> Зберігає довгострокову пам'ять агентів, будує граф знань (GraphRAG) на основі завантажених документів.

## Основні компоненти

### 1. Frontend (Клієнтська частина)
- **Фреймворк:** Vue 3 + Vite.
- **Управління станом:** Використовуються реактивні компоненти Vue (Composition API).
- **Стилізація:** CSS/SCSS (ізольовані стилі компонентів).
- **Особливості:** Інтерфейс розбитий на 5 логічних кроків (Сходинок), які відображають життєвий цикл симуляції. Використовуються Server-Sent Events (SSE) для отримання потоків статусів під час генерації звіту та симуляції.

### 2. Backend (Серверна частина)
- **Фреймворк:** Flask (Python). Забезпечує RESTful API.
- **Оркестратор агентів:** OASIS (адаптований фреймворк від CAMEL-AI). Відповідає за життєвий цикл кожного окремого агента, планування їхніх дій та взаємодію.
- **Модель обробки:** Асинхронне виконання задач. Під час симуляції кожен агент працює паралельно з іншими.
- **Двоплатформна симуляція:** Внутрішня логіка імітує дві соціальні платформи:
  - *Платформа 1 (Аналог Twitter/Стрічка):* Швидкий обмін короткими повідомленнями.
  - *Платформа 2 (Аналог Reddit/Форум):* Обговорення за темами (треди), формування інформаційних бульбашок.

### 3. Підсистема пам'яті (Memory & Knowledge Graph)
- **Zep:** Використовується як основне сховище для векторної бази даних та графа знань (GraphRAG).
- **Процес:**
  1. Завантажені документи розбиваються на фрагменти (chunks).
  2. Zep витягує сутності та зв'язки.
  3. Під час симуляції кожна дія агента (пост, коментар) записується в часову пам'ять графа, що дозволяє Report Agent робити глибокий аналіз (InsightForge).

## Потоки даних

1. **Ініціалізація:** Файли (PDF/TXT) -> Парсинг -> Zep (Вузли та Зв'язки) -> Генерація персон агентів (LLM).
2. **Симуляція:** Персони агентів + Параметри середовища -> OASIS Engine -> Генерація постів/коментарів -> Збереження логів у локальну БД (SQLite/JSON) + Оновлення графа знань у Zep.
3. **Аналіз:** Логи + Граф знань -> Report Agent -> Маркдаун звіт.
