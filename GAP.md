# Поглиблений аналіз прогалин (Deep Gap Analysis)

Після детального дослідження архітектури, кодової бази Frontend та Backend, а також з урахуванням попереднього звіту, було виявлено ряд серйозних прогалин, які критично перешкоджають переходу MiroFish у стан **100% Offline** та **Повної Локалізації**.

## 1. Прогалини локалізації (Frontend)

Хоча ми успішно створили словники перекладів (`uk.json`, `en.json` на 660+ ключів), сама кодова база фронтенду досі не повністю інтегрована з системою `i18n`.

### 1.1. Жорстко закодований текст (Hardcoded UI Strings)
Багато компонентів Vue не використовують функцію перекладу `$t('...')`. Замість цього текст вписаний напряму китайською мовою.
- **`src/views/Process.vue`**: Знайдено десятки рядків з китайським текстом у шаблонах (`<template>`). Наприклад: `图谱构建` (Побудова графа), `实时知识图谱` (Граф у реальному часі), `接口说明` (Опис API).
- **Вплив:** Навіть при виборі української мови в UI, ці сторінки продовжуватимуть відображатися китайською.

### 1.2. Метадані `index.html`
- Теги `<title>MiroFish - 预测万物</title>` та `<meta name="description" ...>` залишаються статичними.
- Атрибут `lang="zh"` у тегу `<html>`.
- **Вплив:** Заголовок вкладки браузера завжди китайський.

## 2. Прогалини автономності (100% Offline)

Система все ще має приховані зовнішні залежності, які призведуть до збоїв у закритих корпоративних мережах без доступу до інтернету.

### 2.1. Зовнішні шрифти (Google Fonts)
- У `index.html` підключаються шрифти через CDN: `https://fonts.googleapis.com` та `https://fonts.gstatic.com`.
- **Вплив:** У мережі без інтернету браузер намагатиметься завантажити ці шрифти, що призведе до тривалого зависання (timeout) сторінки на 30-60 секунд при першому відкритті та порушення типографіки.
- **Рішення:** Завантажити ці шрифти (Inter, JetBrains Mono, Space Grotesk) локально в папку `public/fonts` та змінити CSS.

### 2.2. Залежність від OpenAI за замовчуванням
- У `backend/app/config.py` встановлено: `LLM_BASE_URL = os.environ.get('LLM_BASE_URL', 'https://api.openai.com/v1')`.
- **Вплив:** Якщо користувач помилиться у файлі `.env`, система "тихо" спробує стукати на сервери OpenAI замість того, щоб впасти з попередженням або переключитися на Ollama/Localhost (що очікується в Offline-версії).

## 3. Критична Архітектурна Прогалина (Zep Cloud vs Zep Python)

Скрипт `offline_migration.py` передбачає, що `zep-python` є повною дзеркальною заміною (drop-in replacement) для `zep-cloud`. Це **хибне припущення**, яке зламає бекенд.

### 3.1. Несумісність API GraphRAG
- У файлах `graph_builder.py` та `ontology_generator.py` використовуються імпорти:
  ```python
  from zep_cloud import EpisodeData, EntityEdgeSourceTarget, InternalServerError
  from zep_cloud.external_clients.ontology import EntityModel, EntityText, EdgeModel
  ```
- **Вплив:** Бібліотека відкритого вихідного коду `zep-python` (для Zep Community Edition) має іншу архітектуру класів. Класи `external_clients.ontology` є специфічними для хмарної версії (Zep Cloud). 
- **Ризик:** Навіть якщо ми виправимо імпорти за допомогою regex, локальний Zep сервер може не підтримувати методи `client.graph.add_batch` або `client.graph.set_ontology` у тому вигляді, як цього вимагає `camel-oasis` або наш бекенд. 

## Висновок та План дій

Для досягнення поставлених цілей необхідно:
1. **Frontend:** Пройтися по всіх `.vue` файлах та замінити хардкод (`[\u4e00-\u9fa5]+`) на виклики `{{ $t('key') }}`.
2. **Offline Assets:** Видалити Google Fonts з `index.html` і додати їх як локальні ассети.
3. **Backend / Zep:** Провести аудит сумісності `zep-python` API для GraphRAG. Якщо Zep Community не підтримує `external_clients.ontology`, необхідно буде адаптувати логіку графа або змінити провайдера пам'яті для локальної версії.
