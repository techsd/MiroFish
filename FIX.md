# 🛠️ Гайд по хакам, пасхалкам та оптимізаціям для MiroFish (Windows)

Цей посібник містить неочевидні конфігурації, системні "хаки" та оптимізації, які закладені в архітектуру проекту MiroFish і які варто знати при переналаштуванні або розгортанні проекту.

---

## 1. 🤫 Приховані змінні середовища (.env) та LLM оптимізація

У файлі `.env.example` є задокументований, але часто ігнорований "хак" для **прискорення LLM**:

```env
# 加速 LLM 配置（可选）
LLM_BOOST_API_KEY=your_api_key_here
LLM_BOOST_BASE_URL=your_base_url_here
LLM_BOOST_MODEL_NAME=your_model_name_here
```
**Як це працює:** Ви можете використовувати дешеву, але повільну модель (наприклад, базовий Qwen або GPT-4o-mini) для стандартних завдань, але для критичних операцій (таких як OASIS симуляції) підключити швидшу/потужнішу модель через `LLM_BOOST_*`. Якщо ви не використовуєте цю фічу, **видаліть ці рядки** з вашого `.env`, інакше система може намагатися їх підхопити.

## 2. 🪟 Хаки для Windows-середовища (Backend)

У файлі `backend/run.py` реалізовано важливий хак для обходу типової проблеми Windows — "крякозябри" (неправильне кодування) в консолі:

```python
# 解决 Windows 控制台中文乱码问题：在所有导入之前设置 UTF-8 编码
if sys.platform == 'win32':
    os.environ.setdefault('PYTHONIOENCODING', 'utf-8')
    if hasattr(sys.stdout, 'reconfigure'):
        sys.stdout.reconfigure(encoding='utf-8', errors='replace')
```
**Оптимізація:** Якщо ви запускаєте бекенд через нестандартні термінали (наприклад, старий CMD), цей блок гарантує, що кирилиця, ієрогліфи та інші UTF-8 символи не "вб'ють" процес виводу логів.

## 3. 🧹 "Тихі" логи та боротьба з мультипроцесингом

Якщо ви використовуєте важкі бібліотеки (наприклад, Camel AI або Transformers), вони генерують багато спаму. У `backend/app/__init__.py` розробники додали пасхалку-глушник:

```python
# 抑制 multiprocessing resource_tracker 的警告
warnings.filterwarnings("ignore", message=".*resource_tracker.*")
```
**Порада:** Якщо ви розробляєте нові фічі і бачите дивні помилки пам'яті (resource_tracker), закоментуйте цей рядок для детального дебагу. У продакшені це залишається включеним.

А також, для відображення не-ASCII символів в JSON-відповідях Flask (щоб API повертало читабельний текст, а не `\uXXXX`), використовується:
```python
app.json.ensure_ascii = False
```

Якщо `docker-compose.yml` відмовляється завантажувати образ через обмеження або повільний інтернет, всередині є закоментований "хак" з використанням альтернативного дзеркала:

## 5. ⚙️ Контроль бюджету та нескінченних циклів OASIS

Симуляції в MiroFish (OASIS) можуть швидко спалити ваш API-баланс. У `backend/app/config.py` є приховані ліміти, які ви можете перевизначити у вашому `.env`, щоб зупинити "втрату грошей":

```env
# Максимальна кількість раундів симуляції (за замовчуванням 10)
OASIS_DEFAULT_MAX_ROUNDS=10

# Ліміти для Report Agent (щоб він не "думав" вічно)
REPORT_AGENT_MAX_TOOL_CALLS=5
REPORT_AGENT_MAX_REFLECTION_ROUNDS=2
```
**Оптимізація:** Для тестування зменшіть `OASIS_DEFAULT_MAX_ROUNDS` до `2` або `3`.

## 6. ⚡ Zero-Config магія (NPM)

Проект використовує потужну комбінацію в кореневому `package.json`.
Замість того, щоб відкривати два термінали для Frontend та Backend, використовується `concurrently`:

```json
"dev": "concurrently --kill-others -n \"backend,frontend\" -c \"green,cyan\" \"npm run backend\" \"npm run frontend\""
```

А для встановлення абсолютно всього (включно зі створенням віртуального середовища Python через `uv`), достатньо виконати:
```bash
npm run setup:all
```
Це автоматизує розгортання Node.js та Python-пакетів однією командою.

## Підсумок для переналаштування
Якщо ви плануєте серйозні зміни або розгортання:
1. Завжди використовуйте `uv` замість стандартного `pip` (проект оптимізований під нього, див. `uv.lock`).
2. Не використовуйте `localhost` у Windows для бази або API, якщо сидите під WSL, використовуйте `127.0.0.1` або перевіряйте прокидання портів у `vite.config.js`.
3. Слідкуйте за лімітами симуляцій в конфігах, щоб не витрачати токени даремно!
